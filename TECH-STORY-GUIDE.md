# 프롤로그 — “의지는 믿을 수 없으니, 시스템을 세우자”

어느 날 박준영은 깨달았습니다.

**“미래의 나는 오늘의 나와 다른 사람처럼 행동할 수 있다.”**

오늘은 “푸시업 100개를 하겠다”고 말하지만, 내일은 “오늘 컨디션이 별로니까 예외로 하자”고 말할 수 있습니다. 종이에 적으면 지울 수 있고, 혼자 기억하면 기억을 바꿀 수도 있습니다.

그래서 평범한 할 일 앱이 아니라 **디지털 감시탑**을 만들기로 했습니다.

이 감시탑의 임무는 네 가지였습니다.

1. **약속을 기록한다.**
2. **증거가 없으면 성공으로 인정하지 않는다.**
3. **기록을 여러 기기에서 똑같이 보여준다.**
4. **영상은 검사하되 원본은 보관하지 않는다.**

🔴 **공학의 출발점은 코딩이 아니라 “무엇을 해결할 것인가?”를 정확히 정하는 일입니다.**

이것을 **요구사항 정의**라고 합니다.

# 1장 — 제품 설계: 기능보다 먼저 규칙을 세우다

감시탑을 짓기 전에 먼저 법을 정했습니다.

- 하루는 **다음 날 오전 3시**에 끝난다.
- 증빙이 없는 기록은 성공이 아니다.
- 제출한 계획은 마음대로 바꿀 수 없다.
- 비상 루틴은 **20초 이완 → 3분 영상 확인 → 10분 집중** 순서다.

이 단계는 코딩보다 중요합니다. 컴퓨터는 스스로 “좋은 기준”을 만들지 못합니다. 사람이 준 규칙을 매우 빠르고 정확하게 반복할 뿐입니다.

🟢 **컴퓨터는 똑똑한 판사가 아니라, 규칙을 한 글자도 잊지 않는 심부름꾼입니다.**

## 공대생이 기억할 것

🔴 **애매한 문장은 애매한 프로그램을 만듭니다.**

“운동을 열심히 한다”는 코딩하기 어렵습니다.

“푸시업 증빙이 인정된 기록의 합이 하루 100개 이상이다”는 코딩할 수 있습니다.

좋은 요구사항에는 보통 다음이 있습니다.

- **입력:** 무엇을 받는가?
- **처리:** 어떤 규칙으로 판단하는가?
- **출력:** 무엇을 보여주는가?
- **실패 조건:** 언제 실패인가?
- **완료 조건:** 어디까지 되면 끝인가?

# 2장 — React: 화면을 레고 블록으로 만들다

감시탑의 앞문은 React로 만들었습니다.

React는 화면을 한 덩어리로 만들지 않습니다. 작은 부품으로 나눕니다.

이 프로젝트에는 다음과 같은 부품이 있습니다.

- `Today`: 오늘 상태
- `Emergency`: 비상 루틴
- `RecordForms`: 기록 입력
- `Review`: 증빙 검토
- `Today`: 푸시업·윗몸일으키기 오늘 현황
- `Blog`: 실패 회고
- `AuditLog`: 변경 이력
- `Timer`: 20초·10분 타이머

🟢 자동차를 생각하면 쉽습니다.

- 핸들
- 속도계
- 브레이크
- 좌석

각 부품은 따로 만들지만, 합치면 자동차가 됩니다. React의 **컴포넌트**도 같습니다.

## 상태란 무엇일까?

사용자가 비상 루틴 버튼을 눌렀다고 합시다.

화면은 지금 단계가 무엇인지 기억해야 합니다.

```ts
stage: "relax" | "video" | "focus" | "after" | "complete" | "failed"
```

이것을 **상태**라고 합니다.

🔵 `relax`는 20초 이완 단계입니다.  
🔵 `video`는 3분 영상 확인 단계입니다.  
🔵 `focus`는 10분 집중 단계입니다.  
🔵 `complete`는 성공입니다.  
🔵 `failed`는 실패입니다.

🔴 **화면은 상태를 보여주고, 사용자의 행동은 상태를 바꿉니다.**

이 한 문장을 이해하면 React의 절반을 이해한 것입니다.

## 이벤트란 무엇일까?

버튼 클릭, 글자 입력, 파일 선택은 모두 **이벤트**입니다.

```tsx
<button onClick={() => signOut(auth)}>로그아웃</button>
```

뜻은 간단합니다.

> “이 버튼을 누르면 로그아웃해.”

# 3장 — TypeScript: 값에 이름표를 붙이다

JavaScript는 자유롭습니다. 너무 자유로워서 실수도 자유롭게 생깁니다.

TypeScript는 값에 이름표를 붙입니다.

```ts
type Kind = "pushup" | "situp" | "stock";
```

이 말은 기록 종류가 세 가지 중 하나여야 한다는 뜻입니다.

잘못해서 `"pizza"`를 넣으면 개발 중에 경고를 받을 수 있습니다.

🟢 TypeScript는 학교 준비물을 검사하는 선생님과 같습니다.

> “수학 시간인데 왜 프라이팬을 가져왔니?”

## 공대생이 기억할 것

🔴 **타입은 프로그램이 받을 수 있는 값의 범위를 설명하는 문서이자 안전장치입니다.**

타입이 있다고 모든 오류가 사라지는 것은 아닙니다. 하지만 잘못된 값을 훨씬 일찍 발견할 수 있습니다.

# 4장 — Chrome 브라우저: 단순한 창이 아니라 작은 운영체제

Chrome은 웹페이지를 보여주는 창만이 아닙니다.

Chrome 안에는 다음 능력이 들어 있습니다.

- JavaScript 실행
- 파일 선택
- 영상 재생
- 픽셀 분석
- 암호 계산
- 캐시 저장
- 인터넷 요청
- 로그인 팝업

그래서 영상 원본을 서버에 보내지 않고도 브라우저 안에서 검사할 수 있었습니다.

## DOM

Chrome은 HTML을 읽고 화면 요소를 나무처럼 정리합니다. 이것이 **DOM**입니다.

```text
화면
├─ 제목
├─ 메뉴
├─ 기록 입력칸
└─ 저장 버튼
```

JavaScript는 이 나무의 가지를 찾고 내용을 바꿉니다.

## 브라우저의 출처(Origin)

브라우저는 주소가 다르면 다른 집으로 취급합니다.

```text
https://hamster9212.github.io
https://stikk-referee.web.app
```

두 주소는 서로 다른 집입니다. 한 집의 `localStorage`를 다른 집이 마음대로 읽을 수 없습니다.

🟡 **이 보안 원리 때문에 GitHub Pages에 저장된 과거 로컬 기록은 Firebase Hosting 주소에서 자동으로 읽을 수 없습니다.**

🔴 **웹 보안에서 “주소가 다르다”는 것은 단순한 이름 차이가 아니라 권한 경계가 다르다는 뜻입니다.**

# 5장 — 영상 분석: 영상을 보관하지 않고 검사하다

비상 루틴에는 3분 영상 확인이 필요했습니다. 하지만 영상 원본은 저장하고 싶지 않았습니다.

해결 방법은 다음과 같습니다.

1. 사용자가 영상 파일을 선택한다.
2. 브라우저가 파일을 임시로 읽는다.
3. 영상 길이를 확인한다.
4. 5초 간격으로 장면을 가져온다.
5. 중앙 부분의 픽셀 변화를 계산한다.
6. 판정 결과와 해시만 남긴다.
7. 임시 영상 주소를 해제한다.
8. 파일 입력칸을 비운다.

## `URL.createObjectURL`

브라우저는 사용자가 선택한 파일을 가리키는 임시 주소를 만듭니다.

```ts
const url = URL.createObjectURL(file);
```

검사가 끝나면 다음 코드로 임시 주소를 없앱니다.

```ts
URL.revokeObjectURL(url);
```

🔴 **원본 영상은 Firebase에 업로드되지 않습니다. 브라우저가 판정하는 동안에만 읽습니다.**

## Canvas와 픽셀

영상의 특정 장면을 작은 Canvas에 그립니다.

```ts
canvas.width = 96;
canvas.height = 96;
```

그다음 중앙 영역의 픽셀을 읽습니다.

```ts
ctx.getImageData(24, 24, 48, 48)
```

전체 96×96 화면 중 중앙 48×48 부분을 보는 것입니다.

이 부분이 **ROI**입니다.

🟢 운동장에서 특정 선수 한 명만 확대해서 보는 것과 같습니다.

## 평균과 분산

픽셀 밝기의 평균을 구하면 장면이 전체적으로 얼마나 밝은지 알 수 있습니다.

분산을 구하면 픽셀들이 얼마나 서로 다른지 알 수 있습니다.

- 분산이 작다: 화면이 거의 한 색이다.
- 분산이 크다: 모양과 경계가 보인다.

하지만 중요한 한계가 있습니다.

🟡 **현재 방식은 실제 호흡을 의학적으로 측정하지 않습니다. 중앙 영상 영역의 반복적인 밝기 변화만 검사합니다.**

따라서 이것은 의료 기기가 아니라 **행동 증빙을 돕는 보수적인 규칙 검사기**입니다.

# 6장 — SHA-256: 파일의 디지털 지문

같은 영상을 반복해서 제출하는 것을 막으려면 파일을 비교해야 합니다.

파일 전체를 서버에 보관할 필요는 없습니다. 대신 **해시**를 계산합니다.

```ts
crypto.subtle.digest("SHA-256", bytes)
```

결과는 다음처럼 긴 글자입니다.

```text
6f1ed0...아주 긴 값...
```

파일 내용이 조금만 달라져도 결과가 크게 달라집니다.

🟢 사람은 지문으로 같은 사람인지 확인합니다.  
🟢 프로그램은 해시로 같은 파일인지 확인할 수 있습니다.

## 꼭 알아야 할 한계

🔴 **해시는 암호화가 아닙니다.**

- 암호화: 열쇠가 있으면 원문으로 되돌릴 수 있습니다.
- 해시: 보통 원문으로 되돌리는 용도가 아닙니다.

또한 해시가 있다고 파일 내용이 좋은지 판단할 수 있는 것은 아닙니다. 해시는 주로 **동일성 확인**에 사용합니다.

# 7장 — Firebase Authentication: Google이 신분증을 확인하다

직접 비밀번호를 보관하면 위험합니다.

그래서 앱은 Google 로그인을 사용합니다.

흐름은 다음과 같습니다.

```text
사용자
  ↓ Google 로그인 버튼
Google
  ↓ 신원 확인
Firebase Authentication
  ↓ UID 발급
우리 앱
```

앱은 사용자의 Google 비밀번호를 받지 않습니다.

🔴 **Google 비밀번호는 우리 앱 코드와 Firestore에 저장되지 않습니다.**

## OAuth의 핵심

OAuth는 비밀번호를 다른 앱에 넘기지 않고 권한을 확인하는 방식입니다.

놀이공원 손목띠를 생각하면 쉽습니다.

- 매번 주민등록증을 보여주지 않습니다.
- 입구에서 한 번 확인합니다.
- 이후에는 손목띠로 입장 권한을 증명합니다.

Firebase에서는 사용자마다 **UID**를 줍니다.

이 UID가 데이터 창고의 방 번호가 됩니다.

---

# 8장 — Firestore: 사용자마다 잠긴 기록 방

Firestore는 자료를 문서 모양으로 저장합니다.

이 앱의 핵심 경로는 다음과 같습니다.

```text
users/{userId}
```

예를 들어 UID가 `abc123`이면 다음 방에 기록합니다.

```text
users/abc123
```

다른 사용자는 다른 방을 받습니다.

## 실시간 동기화

앱은 `onSnapshot`으로 Firestore의 변화를 지켜봅니다.

```ts
onSnapshot(userRef, snapshot => {
  // 새 기록을 화면에 반영
});
```

컴퓨터에서 기록을 바꾸면 Firestore가 알려주고, 휴대전화 화면도 새 자료를 받습니다.

🟢 단체 채팅방에 새 메시지가 오면 모든 사람 화면에 나타나는 원리와 비슷합니다.

🔴 **여러 기기 동기화는 “기기끼리 직접 연결”하는 것이 아니라, 모두 같은 클라우드 원장을 바라보는 방식입니다.**

## 오프라인 캐시

Firestore는 최근 자료를 브라우저에 캐시할 수 있습니다.

인터넷이 잠시 끊기면:

1. 최근 기록을 가까운 캐시에서 보여줍니다.
2. 변경 사항을 대기시킵니다.
3. 인터넷이 돌아오면 서버와 맞춥니다.

🟡 동기화 전에 브라우저 자료를 지우면 아직 서버로 가지 않은 변경은 사라질 수 있습니다.

# 9장 — 보안 규칙: 문 앞을 지키는 경비원

로그인만 만들고 데이터베이스를 아무나 읽게 두면 큰 사고가 납니다.

그래서 다음 규칙을 배포했습니다.

```txt
allow read, create, update:
  if request.auth != null
  && request.auth.uid == userId;

allow delete: if false;
```

쉬운 뜻은 다음과 같습니다.

1. 로그인한 사람만 들어올 수 있다.
2. 로그인 UID와 방 번호가 같아야 한다.
3. 삭제 요청은 거부한다.

🔴 **화면에서 버튼을 숨기는 것은 보안이 아닙니다. 서버가 요청을 거부해야 진짜 보안입니다.**

## Firebase 설정값은 왜 공개 코드에 있을까?

`apiKey`, `projectId`, `authDomain` 같은 Firebase 웹 설정값은 앱이 어느 Firebase 프로젝트를 사용할지 알려주는 표지판입니다.

이 값만으로 모든 데이터를 읽을 수 있는 것은 아닙니다. 실제 출입 통제는 로그인과 보안 규칙이 담당합니다.

하지만 다음은 절대로 공개하면 안 됩니다.

- 서비스 계정 비밀키
- GitHub 개인 토큰
- 관리자 비밀번호
- 결제 API 비밀키

🟢 Firebase 웹 설정값은 건물 주소입니다.  
🔴 서비스 계정 키는 건물의 마스터키입니다.

# 10장 — npm과 의존성: 다른 기술자의 부품을 가져오다

모든 기능을 처음부터 만들면 너무 오래 걸립니다.

그래서 검증된 도구를 가져옵니다.

```json
"firebase": "^12.16.0",
"react": "19.2.6",
"typescript": "5.9.3"
```

이런 외부 도구를 **의존성**이라고 합니다.

`package.json`은 필요한 부품 목록입니다.

`package-lock.json`은 실제 설치된 부품의 정확한 버전을 기록합니다.

## npm 명령어

```bash
npm ci
```

잠금 파일에 적힌 버전 그대로 깨끗하게 설치합니다.

```bash
npm run build
```

배포용 파일을 만듭니다.

```bash
npm test
```

빌드와 자동 테스트를 실행합니다.

🔴 **외부 라이브러리는 시간을 아껴주지만, 보안 문제와 버전 충돌도 함께 가져올 수 있습니다.**

그래서 업데이트, 보안 감사, 테스트가 필요합니다.

# 11장 — Vite와 빌드: 설계도를 실제 건물로 바꾸다

개발할 때 작성한 TypeScript와 React 코드는 그대로 배포되지 않습니다.

Vite가 다음 일을 합니다.

1. 코드를 읽는다.
2. 브라우저용 JavaScript로 바꾼다.
3. 여러 파일을 묶는다.
4. 불필요한 부분을 줄인다.
5. `dist/client` 폴더에 결과물을 만든다.

이것이 **빌드**입니다.

```ts
build: {
  outDir: "dist/client"
}
```

🟢 설계도와 자재를 실제 건물 부품으로 조립하는 공장과 같습니다.

## BASE_PATH 문제

GitHub Pages에서는 앱이 다음 하위 경로에 있었습니다.

```text
/stikK-/
```

Firebase Hosting에서는 루트 `/`에 있습니다.

그래서 빌드 주소를 바꿀 수 있게 만들었습니다.

```ts
base: process.env.BASE_PATH || "/stikK-/"
```

Firebase용 빌드는 다음처럼 실행했습니다.

```bash
BASE_PATH=/ npm run build
```

🔴 **배포 주소가 달라지면 코드가 파일을 찾는 기준 주소도 달라질 수 있습니다.**

이 원리를 놓치면 CSS나 JavaScript 파일이 404로 사라집니다.

# 12장 — 404 사건: 파일은 있는데 주소를 못 찾다

처음 공개 앱에서 404가 발생했습니다.

404는 서버가 이렇게 말하는 것입니다.

> “요청한 주소에서 파일을 찾지 못했습니다.”

가능한 원인은 다양합니다.

- 빌드 결과물이 없음
- 배포 폴더가 잘못됨
- 하위 경로 설정이 잘못됨
- SPA 주소를 서버가 실제 파일로 착각함

Firebase Hosting에는 다음 규칙이 있습니다.

```json
"rewrites": [
  { "source": "**", "destination": "/index.html" }
]
```

뜻은 다음과 같습니다.

> “어떤 앱 주소로 들어오더라도 우선 `index.html`을 보여줘.”

그다음 React가 알맞은 화면을 담당합니다.

🔴 **SPA에서는 브라우저 주소와 서버의 실제 파일 위치가 다를 수 있습니다.**

# 13장 — Git과 GitHub: 시간 여행이 가능한 코드 기록

코드를 바꾸다 실패하면 이전 상태로 돌아가야 합니다.

Git은 변경 이력을 저장합니다.

```text
코드 수정
  ↓
git add
  ↓
git commit
  ↓
git push
  ↓
GitHub
```

## 중요한 낱말

- `add`: 다음 커밋에 넣을 변경을 고릅니다.
- `commit`: 변경에 이름을 붙여 저장합니다.
- `push`: 커밋을 GitHub에 보냅니다.
- `main`: 보통 배포 기준이 되는 중심 가지입니다.

이 프로젝트에서는 다음과 같은 커밋이 만들어졌습니다.

- 비상 루틴 추가
- 타이머 새로고침 복구
- Firebase 동기화
- Firebase Hosting 설정

🔴 **좋은 커밋은 “무엇을 왜 바꿨는지”를 한 덩어리로 설명합니다.**

# 14장 — GitHub Actions: 로봇 배포 직원

GitHub에 새 코드가 올라오면 자동 배포 작업이 실행됩니다.

이 자동 로봇을 **GitHub Actions**라고 합니다.

로봇은 다음 순서로 일합니다.

1. 코드를 내려받는다.
2. Node.js를 준비한다.
3. 의존성을 설치한다.
4. 앱을 빌드한다.
5. 결과물을 업로드한다.
6. GitHub Pages에 배포한다.

이것을 **CI/CD**라고 부릅니다.

- CI: 코드가 합쳐져도 정상인지 계속 검사합니다.
- CD: 통과한 결과를 자동으로 배포합니다.

🟢 사람이 매번 같은 요리를 하지 않고 자동 조리 기계를 만든 것과 같습니다.

# 15장 — Firebase Hosting과 Cloud Shell

GitHub Pages 주소는 Firebase 로그인 승인 과정에서 문제가 생겼습니다.

그래서 Firebase가 기본으로 신뢰하는 주소에 앱을 배포했습니다.

```text
https://stikk-referee.web.app
```

Cloud Shell에서 실행한 핵심 명령은 다음과 같습니다.

```bash
git clone https://github.com/hamster9212/stikK-.git
cd stikK-
npm ci
BASE_PATH=/ npm run build
firebase deploy --only hosting,firestore:rules --project stikk-referee
```

## 각 줄의 뜻

🔵 `git clone`: GitHub 코드를 복사합니다.  
🔵 `cd`: 작업 폴더로 이동합니다.  
🔵 `npm ci`: 필요한 도구를 정확한 버전으로 설치합니다.  
🔵 `npm run build`: 배포 파일을 만듭니다.  
🔵 `firebase deploy`: Firebase에 결과물과 보안 규칙을 보냅니다.

🔴 **CLI 명령은 마법 주문이 아닙니다. 파일을 복사하고, 변환하고, 서버로 보내는 작업을 짧게 표현한 문장입니다.**

# 16장 — HTTPS, CDN, 캐시: 사용자의 눈에 보이지 않는 고속도로

배포가 끝나면 Firebase Hosting은 HTTPS 주소를 제공합니다.

## HTTPS

사용자와 서버 사이의 통신을 암호화합니다.

카페 Wi-Fi를 사용하더라도 중간 사람이 내용을 쉽게 훔쳐보지 못하게 합니다.

## CDN

사용자에게 가까운 서버가 웹파일을 전달합니다.

한국 사용자가 미국 서버까지 매번 가지 않아도 가까운 지점에서 파일을 받을 수 있습니다.

## 캐시

이미 받은 파일을 다시 사용할 수 있게 저장합니다.

빠르지만 오래된 파일이 남는 문제가 생길 수 있습니다.

🟡 새 버전이 배포됐는데 옛 화면이 보이면 새로고침이나 캐시 갱신이 필요할 수 있습니다.

---

# 17장 — Service Worker: 오프라인을 돕는 작은 직원

Service Worker는 웹페이지와 별도로 브라우저 뒤에서 움직입니다.

이 프로젝트에서는 네트워크 요청이 성공하면 응답을 캐시에 복사하고, 인터넷 연결이 실패하면 캐시를 사용합니다.

```text
인터넷 요청
├─ 성공 → 화면에 전달 + 캐시에 복사
└─ 실패 → 캐시에서 찾기
```

하지만 Service Worker는 강력한 만큼 주의해야 합니다.

- 오래된 파일을 계속 보여줄 수 있습니다.
- 잘못된 캐시 규칙은 업데이트를 방해합니다.
- HTTPS 환경에서 사용해야 합니다.

🔴 **오프라인 기능은 “인터넷이 없어도 모든 것이 된다”는 뜻이 아닙니다. 로그인이나 최초 데이터 다운로드는 인터넷이 필요할 수 있습니다.**

# 18장 — 타이머 설계: 숫자를 줄이는 것보다 종료 시각을 저장하라

단순 타이머는 매초 숫자를 하나씩 줄일 수 있습니다.

하지만 새로고침하면 숫자가 사라집니다.

더 튼튼한 방법은 **시작 시각 또는 종료 시각을 저장하는 것**입니다.

```text
남은 시간 = 종료 시각 - 현재 시각
```

이렇게 하면 브라우저가 잠시 멈추거나 새로고침되어도 실제 시간이 흐른 만큼 계산할 수 있습니다.

🔴 **시간 기반 시스템은 “몇 번 감소했는가”보다 “실제 시각이 언제인가”를 기준으로 설계하는 편이 안전합니다.**

# 19장 — 테스트와 Lint: “내 컴퓨터에서는 돼요”를 막는 장치

프로그램은 한 번 실행된다고 완성된 것이 아닙니다.

이 프로젝트에서는 다음을 검사했습니다.

- 빌드 성공 여부
- 코드 규칙 위반
- 주간 달성률이 100%를 넘지 않는지
- 인정된 푸시업과 윗몸일으키기를 각각 더하는지
- 같은 이미지 해시를 찾는지
- 검토 중 기록을 확정하지 않는지

## 테스트의 기본 모양

```text
준비
→ 실행
→ 예상 결과와 비교
```

이를 영어로 자주 다음처럼 부릅니다.

- Arrange: 준비
- Act: 실행
- Assert: 확인

🔴 **테스트는 버그가 없음을 증명하지 않습니다. 하지만 이미 약속한 기능이 다시 망가지는 것을 빠르게 알려줍니다.**

## Lint

Lint는 오탈자, 위험한 코드 모양, 일관성 문제를 검사합니다.

문법 시험과 글쓰기 맞춤법 검사를 합친 것과 비슷합니다.

# 20장 — 이 시스템의 정직한 한계

좋은 공학자는 “되는 것”만 말하지 않고 “안 되는 것”도 말합니다.

현재 시스템에는 다음 한계가 있습니다.

1. 영상 분석은 실제 호흡을 의학적으로 측정하지 않습니다.
2. 중앙 픽셀의 반복 변화로 행동을 추정합니다.
3. 조명과 촬영 각도가 판정에 영향을 줄 수 있습니다.
4. 앱 화면은 기록을 잠그지만 프로젝트 관리자 권한까지 기술적으로 제거하지는 않습니다.
5. 사용자 문서 전체를 한 번에 저장하므로 여러 기기에서 동시에 수정하면 마지막 저장이 앞선 변경을 덮을 위험이 있습니다.
6. 공개 실패글의 완전 자동 게시에는 별도의 안전한 서버 작업이 필요합니다.
7. GitHub Pages의 과거 로컬 자료는 Firebase 주소와 출처가 달라 자동 이전되지 않습니다.

🔴 **한계를 정확히 말하는 것은 실패가 아니라 공학적 신뢰성입니다.**

# 21장 — 공대생이라면 여기서 더 생각해 볼 문제

## 문제 1: 동시 수정

컴퓨터와 휴대전화가 동시에 같은 기록을 수정하면 어떻게 합칠까요?

가능한 개선:

- 기록마다 별도 문서 사용
- 서버 타임스탬프 사용
- 트랜잭션 사용
- 버전 번호 비교

## 문제 2: 삭제 불가 원장

화면에서 삭제 버튼을 없애는 것만으로 충분할까요?

아닙니다. 서버 규칙, 별도 감사 로그, 관리자 권한 분리까지 생각해야 합니다.

## 문제 3: 더 정확한 영상 판정

밝기 평균 대신 무엇을 쓸 수 있을까요?

- 얼굴의 주요 점 찾기
- 코·입 위치 추적
- Optical Flow로 움직임 추정
- 시간에 따른 주기 분석
- 기기 안에서 실행되는 작은 AI 모델

## 문제 4: 개인정보 최소화

“기술적으로 저장할 수 있다”와 “저장해야 한다”는 다릅니다.

🔴 **가장 안전한 개인정보는 처음부터 수집하지 않은 개인정보입니다.**

이 원칙 때문에 영상 원본을 클라우드에 보내지 않았습니다.

- **제품 요구사항**
- **상태 기반 화면**
- **브라우저 파일 처리**
- **로컬 영상 분석**
- **암호학적 해시**
- **Google 로그인**
- **실시간 클라우드 데이터베이스**
- **사용자별 접근 통제**
- **오프라인 캐시**
- **Git 버전 관리**
- **자동 빌드와 배포**
- **HTTPS 웹 호스팅**
- **테스트와 코드 검사**

그리고 가장 중요한 공학 원리는 이것입니다.

> 🔴 **좋은 시스템은 사람의 기억과 의지를 기대하지 않습니다. 좋은 행동이 자연스럽게 이어지도록 길을 만들고, 잘못된 행동이 어렵도록 문을 잠급니다.**

# 🧠 5분 복습 카드

**Q. React의 상태란?**  
현재 화면이 기억하는 값입니다.

**Q. Firebase Authentication의 역할은?**  
로그인한 사람이 누구인지 확인합니다.

**Q. UID는 어디에 쓰나?**  
사용자마다 다른 데이터 방을 정할 때 씁니다.

**Q. Firestore는 무엇인가?**  
여러 기기에서 함께 볼 수 있는 클라우드 기록 창고입니다.

**Q. 보안 규칙은 왜 필요한가?**  
다른 사용자가 내 기록을 읽거나 바꾸지 못하게 합니다.

**Q. SHA-256은 무엇인가?**  
파일의 동일성을 확인하는 디지털 지문입니다.

**Q. 영상 원본은 어디에 저장되나?**  
클라우드에 저장되지 않습니다. 판정 중 브라우저에서만 읽습니다.

**Q. Vite 빌드는 무엇을 하나?**  
개발 코드를 배포 가능한 웹파일로 바꿉니다.

**Q. Git 커밋은 무엇인가?**  
코드의 특정 상태를 이름 붙여 저장한 사진입니다.

**Q. 배포란?**  
앱을 다른 사람이 인터넷으로 사용할 수 있게 공개하는 일입니다.
